# Sub-project C — Notifications engine (design + plan)

**Date:** 2026-05-30
**Status:** Proposed
**Program:** Camp 404 audit remediation, sub-project **C** — the first of the interconnected C → E → D feature cluster. Introduces a **Drizzle schema migration**, so higher-stakes than A/B.

## Problem (from the audit)

The in-app fan-out works, but only for `scope = 'everyone'`:
1. **Audience resolution is missing.** The `broadcast_scope` enum, `broadcasts.team`, and `broadcast_targets` all exist, but `publishAnnouncement` hardcodes "every real member." `team` / `team_leads` / `drivers` / `individual` scopes — and the `team_message` / `lead_directive` kinds — are dead. "Team-based push notifications" can't be sent.
2. **No scheduling.** `broadcasts` has `publishedAt` + `dispatchedAt` but no `send_at`; publish fans out immediately. There's no "compose now, fire at the phase-unlock timestamp" path.
3. **No delivery dedupe.** `notification_deliveries` has no `(broadcast_id, user_id)` unique constraint; today idempotency rests solely on the inline `publishedAt` draft-claim. A separated/retryable fan-out worker (which scheduling needs) could double-insert.

## Design

### Schema migration (additive, backward-compatible)
- `broadcasts.send_at timestamp NULL` — when the broadcast should fan out. NULL or `<= now` ⇒ immediate.
- Partial unique index `notification_deliveries (broadcast_id, user_id) WHERE broadcast_id IS NOT NULL` — lets the worker `INSERT … ON CONFLICT DO NOTHING`. (System rows keep `broadcast_id` NULL and are exempt.)
- *(Deferred to D:* the `push_status` drain index — no consumer until the push worker exists.)
- Generated via `pnpm --filter @camp404/db db:generate` (offline; no DB needed). Migration file committed alongside `schema.ts`.

### Audience resolution — `resolveAudience(tx, broadcast)` in `packages/db/src/broadcasts.ts`
Returns the recipient user ids for a broadcast, always excluding system actors, sanitised accounts, and the sender:
- `everyone` → all real members (today's behaviour).
- `team` → `team_memberships` where `team = broadcast.team`.
- `team_leads` → `team_memberships` where `is_lead = true` (distinct users).
- `drivers` → `driver_profiles` where `intends_to_drive = true` (the derived "driver" per AGENTS.md).
- `individual` → `broadcast_targets` for this broadcast.

One definition, reused by the inline publish and the scheduled worker.

### Generalised fan-out
- Refactor `publishAnnouncement`'s recipient query to call `resolveAudience`. Behaviour for the existing camp-wide announcement is unchanged (scope stays `everyone`).
- Add a generic `publishBroadcast({ id, senderId })` (or generalise the existing one) that works for any scope/kind — the data-layer primitive E and D build on.
- **Scheduled path:** a broadcast with a future `send_at` stamps `publishedAt` but leaves `dispatchedAt` NULL. A new cron `/api/cron/notifications/dispatch` (using the B `assertCron` helper) materialises deliveries for broadcasts where `published_at IS NOT NULL AND dispatched_at IS NULL AND (send_at IS NULL OR send_at <= now())`, then stamps `dispatched_at`. The dedupe index makes a retry/overlap safe. Register it in `vercel.json` (daily, within the Hobby cap; immediate sends still fan out inline so they don't wait for the cron).

### Types
Extend `@camp404/types` with a `BroadcastScope` enum and a `send_at`-aware compose input for the generic path, **without** changing `ComposeAnnouncementInput` (the camp-wide announcement composer stays as-is; the scoped/scheduled UI is E/D's surface). Add a one-line comment on `ComposeAnnouncementInput` that it intentionally covers only the camp-wide subset.
